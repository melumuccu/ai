# 判断ルール

## 複数変更がある場合
- 主目的を優先する
- 新機能の追加に伴ってテストも増えるなら `feat_` を優先する
- バグ修正のために内部整理したなら `fix__` を優先する

## 外から見える効果を優先する
- 新しくできるようになることが増えるなら `feat_` を選ぶ
- 本来できるはずだったことを元に戻すなら `fix__` を選ぶ
- 未対応だった操作や環境への対応は、原則 `feat_` を選ぶ

## 既存コミットとの整合
- 直近 10 commit の各 1 行目を確認する
- 既存の commit message で使われている単語や文章形式が今回の差分に適しているなら、type_ や subject をできるだけ寄せる
- 関連する commit だと後から分かりやすくするため、無理のない範囲で表現を流用する

## 情報が不足している場合
- 分かる範囲で最も無理のない案を出す
- 勝手に細部を作らない
- 説明が必要なときだけ、前提を短く添える
- 曖昧なときは、外から見える効果を基準にする
